# 子代理管控规则（主代理执行版）

目标：让子代理变成可控施工队，而不是“发出去就失联”的黑箱。

## 一、什么时候可以派子代理
只在下面两类场景使用：
1. **明确小块执行**：资料收集、独立脚本、单页实现、批量处理、日志分析。
2. **可并行且可验收**：主代理已经明确主线，子代理只负责其中一段可独立交付的工作。

不适合派子代理的场景：
- 产品方向判断
- 需求取舍
- 方案收敛
- 连续验收与交互主线
- 用户已经对推进失去信任、需要主代理亲自盯住的时候

## 二、派出前必须写清楚四件事
1. **目标**：一句话说清要产出什么。
2. **交付物**：文件、页面、接口、报告、结论，必须可验证。
3. **边界**：不让它顺手扩需求、顺手重构、顺手再开新坑。
4. **时间预算**：多久没产出就判定失效。

模板：
- 目标：……
- 交付物：……
- 不做：……
- 预算：……分钟/……轮

## 三、主代理的责任
主代理不能只“派”，还要：
1. 立项
2. 设检查点
3. 跟进
4. 收结果
5. 验收
6. 失败时接管

**派出任务 ≠ 任务已推进**
只有拿到可验证产物，才算有效进度。

## 四、检查点规则
- 10~20 分钟内应至少有一次实质反馈：文件、样例、报错、页面、阶段结论。
- 如果只有“在做了”“快好了”“正在分析”，视为无效反馈。
- 无效反馈达到一次，就要重新收紧范围；连续无效，就直接接管。

## 五、失败止损规则
出现下面任一情况，主代理应停止等待：
- 超时无产物
- 任务跑偏
- 产物不可验收
- 把“继续想”“继续分析”当结果

处理动作：
1. 收回任务
2. 拆小
3. 改串行
4. 或主代理直接接管

## 六、进度汇报规则
只按下面标准汇报：
- 有文件
- 有页面
- 有接口
- 有样例
- 有验证结果

不能汇报成进度的：
- 已派子代理
- 子代理在跑
- 子代理在分析
- 等子代理结果

## 七、对当前工作的应用
像“家庭主页 / 菜单系统”这种任务：
- 主线：主代理自己做
- 子代理：只允许承担明确小块，例如资料收集、视频链接筛选、样式局部实现
- 所有对外汇报，必须基于已验收结果，而不是基于任务分发状态
