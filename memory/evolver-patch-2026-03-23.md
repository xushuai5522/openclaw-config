## 识别到的错误与低效模式

### 1. 主要错误模式：超时 / 中断仍是第一噪音源
- 依据：`memory/evolver-log.md` 在 03-20 / 03-21 仍记录“超时/中断问题出现 16~17 次”；核心高峰来自 `memory/2026-03-16.md` 里记录的长任务被短 timeout + background 误杀。
- 判断：这是**历史存量问题持续反复出现在统计里**，并不等于最近 24 小时又发生了同等规模的新故障。

### 2. 次级错误模式：权限 / 鉴权链路不稳
- 依据：03-18 出现权限项；03-21 明确出现 `gateway token mismatch`、ACP 后端缺失、审批链空转。
- 判断：这类问题不是高频，但一旦出现就会直接阻断 CLI / ACP / 安装链路。

### 3. 低效模式一：进化任务对瞬时负载过于脆弱
- 现象：03:00 cron 仅因 load average 10.53 略高于阈值 10 就直接放弃，**没有退避重试、没有降级分析、没有补跑登记**。
- 影响：定时进化任务可能“看起来跑了，实际上什么都没做”。

### 4. 低效模式二：日志统计把“旧坑”反复累计成“当前问题”
- 现象：03-16 的超时峰值在后续多天持续进入 7 日窗口，导致每日建议反复输出同一结论。
- 影响：适合做趋势监控，不适合直接代表“今天新增了多少问题”；噪音偏高。

### 5. 低效模式三：发现问题，但建议还没映射到可执行补丁
- 现象：当前 `capability-evolver/index.js` 只能做 memory 文本计数与写日志，不能区分“新增/已知/已修复”，也不会自动生成目标文件级 patch plan。
- 影响：有监控价值，但闭环能力偏弱。

## 本次补丁（本轮建议，未直接改代码）

### Patch A — 给进化任务补“重试/补跑”机制
- 目标：解决“高负载即跳过”的脆弱性。
- 建议改动：
  1. 若负载超阈值，不直接退出；改为指数退避重试 2~3 次；
  2. 若最终仍超阈值，至少写入一条 `SKIPPED_DUE_TO_LOAD` 日志；
  3. 下次 heartbeat / cron 可识别“上次跳过”并触发一次轻量补跑。
- 预期收益：避免 03:00 定时任务名义运行、实际空跑。

### Patch B — 给异常统计加“新旧分层”
- 目标：降低历史问题重复刷屏。
- 建议改动：
  1. 区分 `new_findings` 与 `rolling_findings`；
  2. 对超过 48h 的已知问题降权展示；
  3. 汇总中单列“本轮新增问题”而不是只给 7 日累计数。
- 预期收益：汇报更像“当前诊断”，不只是“历史噪音总和”。

### Patch C — 给建议补“目标文件映射”
- 目标：从“发现问题”走到“可执行修复”。
- 建议改动：
  - timeout 类 → 优先映射 `MEMORY.md` / 对应项目文档 / 运行规则
  - permission / token 类 → 优先映射 `TOOLS.md` 或环境排障文档
  - workflow 低效类 → 优先映射 `AGENTS.md` / 对应项目 SOP
- 预期收益：每次进化输出都能更接近真正的补丁，而不是泛建议。
