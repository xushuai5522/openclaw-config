## memory 修复过程
### 根因拆解
1. **本地运行时问题**：Gateway 启动时未带 `--experimental-sqlite`，导致 `memory_search` 直接报 `No such built-in module: node:sqlite`。
2. **远程 embedding 配置问题**：`OPENAI_API_BASE` 指向 `https://www.zhongzhuan.win/v1`，出现证书/重定向问题，embedding 请求失败。
3. **本地 embedding 初始化问题**：切到 local provider 后，自动下载模型不走现有代理，导致初始化和建索引卡住。

### 采取的动作
- 为 Gateway 补上 SQLite 运行时支持。
- 修正 memory 的 provider 路线，放弃不稳定的远程 embedding，改走本地 embedding。
- 手动下载本地 embedding 模型到固定路径，并强制重建索引。
- 实测 `openclaw memory search '中文沟通'` 已能返回结果，说明 memory 检索恢复。

## 技术教训
- **memory 故障要分层排查**：先看本地运行时，再看 provider 配置，最后看模型下载/索引构建，不能把所有失败都归因于“memory 坏了”。
- **SSH 排障优先做非交互化**：公钥登录 + 干净 shell 比临时输密码可靠得多，尤其适合自动化连续排障。
- **重载服务会打断当次回复**：动 Gateway 前要预期消息链路短暂中断，并在恢复后主动做收发验证。

## 复盘结论
